—— THE PROBLEM

Ansible Migrations Are Risky Business

Organizations with hundreds of roles and collections face a daunting question:

  • Will this playbook parse on ansible-core 2.19?
  • Are any modules removed, deprecated, or redirected?
  • Do my module arguments match the argspec for the target version?
  • Are there hardcoded credentials hiding in my project?

Without answers, teams discover issues in staging — or worse, production.

—— THE COST

When You Discover Issues Matters

Early (APME in CI)

Developer fixes it in their branch. Merge proceeds. Nobody else knows.

Staging

Deployment blocked. Team context-switches. Sprint velocity drops.

Production

Outage. Incident response. Postmortem. Customer impact. Headlines.

—

The goal: make "production" the rarest row in this table.

—— THE INSIGHT

Parse Once, Validate Everywhere

APME is a multi-validator static analysis platform. It reads your YAML, reasons about structure and module usage, and reports what it finds — without running tasks or executing against target hosts.

Four independent validation backends run in parallel:

  • OPA/Rego — Policy rules for organizational standards
  • Native Python — Structural and semantic checks
  • Ansible Runtime — Argspec validation against real module signatures
  • Gitleaks — 800+ secret patterns, credential detection

—— GET STARTED

Try APME Today

  • Clone the repo: git clone https://github.com/ansible/apme
  • Install: uv tool install tox --with tox-uv
  • Run a check: apme check /path/to/playbook
  • Start the full pod: tox -e up
  • Open the dashboard: http://localhost:8081
